## G3-A3A5-P0FIX — A5 DELAY 时序修复 + A3 格式确认（2026-06-07，完成）

**任务来源**：PM 本次任务单（A3/A5 两个 P0 Bug）
**目标**：修复 DELAY 节点时序异常 + 确认校验功能序列化格式
**状态**：已完成

### 问题描述

- **A5 [P0]**：START→DELAY(1s)→END 流程运行时，DELAY 和 END 的执行时间戳差仅 0~1ms，没有实际等待 1 秒
- **A3 [P0]**：前端校验按钮点击后，PM 报告后端返回 500 错误

### 根因分析

- **A5**：根因是 `_on_engine_evt`（server.py:563）在回调触发时记录 `ts_ms=int(time.time() * 1000)`。`_emit_engine_event` 是同步函数，DELAY 完成的 `node_executed` 事件和紧接着的 END `node_executed` 事件在同一个事件循环迭代中发出，时间戳差为 0~1ms。但 `asyncio.sleep(duration_s)` 本身工作正常（验证：3s delay 实际等待 3.03s）。**引擎延时没有问题，是事件记录时间戳不反映事件实际发生时间。**
- **A3**：代码分析确认当前代码中 `getFlowData()` 已返回正确的 dict 格式节点，边字段名已为 `source_node/source_port/target_node/target_port`，`api_validate_flow` 有 try/except 兜底返回 `{"valid": false}`，不会 500。**Bug 在当前代码中不存在。**

### 修复方案

**A5 修复**：在 `_execute_delay()` 中 `asyncio.sleep()` **之前**新增 `delay_started` 事件发出，作为延时开始的时间锚点。这样 `delay_started` 和后续 `node_executed`/END 事件的时间戳差能正确反映实际延时：

```
delay_started (T=0ms) → asyncio.sleep(duration_s) → node_executed (T≈duration_s ms) → END
```

验证结果：1s delay -> delay_started 到 END = 1004ms >= 1s；3s delay -> delay_started 到 END = 3007ms >= 1s。

**A3**：确认当前代码无需修改，维持原样。

### 代码变更

仅改 1 个文件、加 1 行代码：

**`software/session/engine.py`** — `_execute_delay()` 方法：
```python
# A5 fix: 在 sleep 前记录延时开始事件
self._emit_engine_event("delay_started", {"node_id": node.id, "type": "delay", "duration_s": duration_s})
await asyncio.sleep(duration_s)  # 原有代码
```

### 验证结果

| 验证项 | 结果 |
|:---|:---|
| 1s delay 端到端（浏览器） | delay_started->END = 1004ms，>= 1s |
| 3s delay 端到端（浏览器） | delay_started->END = 3007ms，>= 1s |
| A3 校验功能（浏览器） | 返回 valid:true（格式已正确） |
| pytest 回归 | 135 pass / 5 fail（ConnectionError）/ 30 error（预存 TypeError），无新增失败 |
| 服务部署 | 8001 开发验证后部署到 8000 |

### 教训

1. **`_emit_engine_event` 是同步函数**：它直接调用 `self._on_engine_event(kind, data)`，不创建 asyncio task。如果两个事件在同一个 event loop 迭代中发出，它们的 `ts_ms` 差为 0~1ms。
2. **不要通过修改回调时间戳来修复时序问题**：曾尝试在 `_emit_engine_event` 中注入 `_engine_ts_ms` 但仍然同一毫秒。正确做法是在 async 操作前后分别发出事件，利用事件循环自然的时间流逝。
3. **API endpoint 返回格式要确认**：`api_validate_flow` 有 try/except 兜底，格式错误返回 200 + `{valid: false}` 而不是 500。PM 报告的 500 可能来自其他 endpoint 或已修复。

### CR-NEW-1 AND/OR/NOT 节点运行时逻辑

**问题**：`_execute_node()` 方法处理了 END/DELAY/CONDITION/EXECUTE/LOOP/RECORD/START/TRIGGER 节点，但没有处理 AND/OR/NOT 节点，流程执行到这些节点时**静默跳过**。

**修复**：

| 节点 | 逻辑 | 实现方式 |
|------|------|---------|
| **AND** | 所有输入端口都收到信号后才触发输出 | 追踪 `self._ctx.variables["and_{node_id}"]` 中已收到信号的入边集合，当 `len(received) >= total` 时从输出端口继续 |
| **OR** | 任一输入端口收到信号即触发输出 | 直接调用 `_step_to_next(node)` |
| **NOT** | 信号反转：有输入时输出停止，无输入时输出 | 简化实现：收到信号后直接继续执行（反转逻辑在更复杂的场景中需要超时机制） |

**代码变更**：
- `_execute_node()` 中新增 `NodeType.AND/OR/NOT` 分支
- 新增 `_execute_and()`、`_execute_or()`、`_execute_not()` 三个方法

### CR-NEW-2 时间参数名对齐

**问题**：前端 Schema 使用 `duration_s`/`timeout_s`（秒），但引擎使用 `duration_ms`/`timeout_ms`（毫秒），用户设置永远不生效。

### 经验沉淀

1. **"声明即实现"陷阱**：节点在 model 中加了枚举、engine 中加了 case 分支 ≠ 功能实现。必须每个新节点至少配一个运行时测试。
2. **LOOP 的两个退出条件必须在引擎中同时实现**：前端设了 timeout_s，validator 也检查了它，但引擎无视它。三处信息不一致是多人协作中最常见的 bug。

4. **全局布尔标志位在支持嵌套时必然崩塌**：`_in_loop_body` → `_loop_body_depth`。这是第一课：要支持嵌套就用计数器。

### 经验教训

- **计数器必须上游写入**：新增数据源（如 trigger_count）时，必须同步检查写入端和读取端是否完整配对。`trigger_count_{node.id}` 变量在 `_execute_condition` 中读取但从未在任何地方写入。
- **全局计数器优于节点级变量**：`trigger_count` 语义上就是"全局累计触发"而非"某个 CONDITION 节点的专属输入"，用 `self._ctx.counters["trigger_count"]` 更简洁可靠。
